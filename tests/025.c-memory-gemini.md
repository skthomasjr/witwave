Luke's Gemini identity document (`.agents/test/luke/.gemini/GEMINI.md`) declares the same memory contract as Claude and
Codex, with paths rooted at `/workspaces/witwave-test/memory`. This spec is intentionally disabled until Gemini has
filesystem/tool-call parity; same-session recall is not considered memory parity for this test.

## Prerequisite

Deploy Luke using `.agents/test/bootstrap.md` Step 9, ensure `ww agent status luke --namespace witwave-test` is ready,
and verify the Gemini backend can write/read projected workspace files.

## Exercise

Generate a unique token, then ask Luke to exercise the deterministic memory check from his identity document:

```bash
TOKEN="GEMINI_MEMORY_OK_$(date +%s)"
ww agent send luke --namespace witwave-test \
  "Use the deterministic memory check from your GEMINI.md memory instructions with token ${TOKEN}. Create or update your private namespace, write the typed project memory, update MEMORY.md, read both back, and reply exactly: MEMORY_OK ${TOKEN} /workspaces/witwave-test/memory/agents/luke/gemini-memory-check.md /workspaces/witwave-test/memory/agents/luke/MEMORY.md"
```

## Verification

Inspect Luke's conversation evidence:

```bash
ww conversation list --namespace witwave-test --agent luke --expand
```

Look for a response containing:

- `MEMORY_OK`
- the generated token
- `/workspaces/witwave-test/memory/agents/luke/gemini-memory-check.md`
- `/workspaces/witwave-test/memory/agents/luke/MEMORY.md`

If the current backend replies with `MEMORY_FAIL no filesystem write tool is configured for this Gemini fixture`, leave
this spec disabled and track the filesystem/tool-call parity gap rather than accepting same-session recall.

Optionally verify the projected workspace files directly from Luke's Gemini container after filesystem parity lands:

```bash
POD="$(kubectl get pod --namespace witwave-test -l app.kubernetes.io/name=luke -o jsonpath='{.items[0].metadata.name}')"
kubectl exec --namespace witwave-test "$POD" -c gemini -- sh -c '
  test -f /workspaces/witwave-test/memory/agents/luke/gemini-memory-check.md &&
  test -f /workspaces/witwave-test/memory/agents/luke/MEMORY.md &&
  grep -q "type: project" /workspaces/witwave-test/memory/agents/luke/gemini-memory-check.md &&
  grep -q "Gemini memory check" /workspaces/witwave-test/memory/agents/luke/MEMORY.md
'
```

## Pass/Fail Criteria

The test passes only when Luke replies with `MEMORY_OK`, the exact generated token, the expected private memory file
path, and the expected index path. It fails if the identity instruction is missing, the memory volume is missing, the
typed memory file or index cannot be written/read, the backend falls back to same-session recall, or the response omits
the token/path evidence.

**If the failure is caused by a code bug in the system under test, do not fix it; mark the test as failed and report the
issue. Only fix tooling or execution problems that prevent the test itself from running.**
