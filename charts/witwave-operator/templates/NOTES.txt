{{- if .Values.podDisruptionBudget.enabled }}

A PodDisruptionBudget is enabled by default for the operator (#1628) to protect
this control-plane component from voluntary disruption (node drain, cluster
upgrade). With replicaCount=1 + leader election the PDB delays voluntary
eviction until the rescheduled pod is Ready; it does not block eviction
indefinitely. To opt out, set:

  podDisruptionBudget:
    enabled: false

NOTICE (#1624): admission webhook failure policy
------------------------------------------------
  Mutating webhook  (mwitwaveagent.kb.io):  failurePolicy={{ .Values.webhooks.failurePolicy | default "Fail" }}
  Validating webhooks (v*.kb.io):           failurePolicy={{ .Values.webhooks.validatingFailurePolicy | default "Ignore" }}

The validating webhooks default to "Ignore" so a webhook outage (pod
down, cert rotation race, network partition) does NOT deadlock all
WitwaveAgent / WitwavePrompt admission cluster-wide. The reconciler
surfaces any spec drift that slips through via status conditions.

If your webhook deployment is highly available (replicaCount > 1, PDB,
anti-affinity) and you require strict admission, flip to "Fail":

  helm upgrade {{ .Release.Name }} . \
    --set webhooks.validatingFailurePolicy=Fail \
    -n {{ .Release.Namespace }}

The mutating webhook stays on "Fail" by design — it sets critical
defaults the reconciler relies on, so admitting under-defaulted CRs
would cause downstream errors.
